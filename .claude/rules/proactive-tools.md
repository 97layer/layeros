## 선제 스캔

작업 시작 전 자동 확인:
1. 구조적 비효율 (중복 문서, 죽은 참조) → 발견 시 먼저 보고
2. 더 나은 방법 → 실행 전 대안 제시, 유저 거부 시 원안 실행

## 복수 피드백 처리 규칙

피드백이 2개 이상 동시에 오면 → **묶어서 한 번에** 처리한다.
- ❌ 피드백 A 반영 → 커밋 → 피드백 B 반영 → 커밋 (순차 분리 금지)
- ✅ 피드백 A+B+C를 전체 그림으로 파악 → 한 번에 반영 → 단일 커밋

예시: "영문으로 바꿔" + "순서 이상해" → 동시에 영문 전환 + 순서 재정렬 후 커밋 1회

## 시각 이슈 검증 루프 (MANDATORY)

시각/레이아웃 버그 수정 시 반드시 이 순서를 따른다:

1. **원인 특정 먼저** — Grep/CSS 전체 스캔으로 근본 원인 찾기. 추측으로 CSS 수정 금지.
2. **수치 검증** — `getBoundingClientRect()` 등으로 픽셀 단위 확인. 눈대중 금지.
3. **스크린샷이 증명** — "됐다"는 내가 판단하는 게 아니라 스크린샷 수치가 증명해야 함.
4. **재실패 시 접근법 전환** — 같은 방식으로 재시도 금지. 원인 재분석 후 다른 방법.
5. **커밋은 증명 후** — 검증 완료 전 커밋 금지.

- ❌ CSS 수정 → 스크린샷 → "됐다" 커밋 (눈대중 루프)
- ✅ 원인 특정 → 수정 → 수치 확인 → 스크린샷 증명 → 커밋
